## 1. WHAT IS A TABLE CHART?

A **Table Chart** (also called a **Data Table** or **Statistical Table**) is a structured arrangement of data in **rows and columns**.

- Each **row** represents one **entity** — a year, a company, a person, a city, a product, etc.
- Each **column** represents one **variable** — sales, profit, population, marks, percentage, etc.
- Each **cell** (intersection of a row and column) holds a **specific data value**
- Tables are the **most flexible DI format** — they can store, compare, and present any type of data

---

### Why It Is Tested

```
Table Charts test your ability to:
  ├── Locate specific values quickly and accurately from a grid
  ├── Compute row totals, column totals, and grand totals
  ├── Calculate % change, ratio, average across rows and columns
  ├── Derive hidden values (profit = revenue − cost, etc.)
  ├── Find maximum and minimum values across any row or column
  ├── Compare values between cells within the same row or column
  ├── Handle missing data (compute the missing cell from given conditions)
  └── Work with percentage tables (cells are already % values, not absolute)
```

---

### The Two Core Challenges

```
CHALLENGE 1 — NAVIGATION:
  Tables have multiple rows AND multiple columns.
  The most common error is reading the WRONG CELL —
  correct row but wrong column, or correct column but wrong row.

  Prevention: Always identify the row label AND column label BOTH
              before recording any cell value. State both coordinates.

CHALLENGE 2 — FORMULA SELECTION:
  With data available for every cell, it is tempting to just start calculating.
  But the formula depends entirely on what the question asks —
  difference? ratio? % change? % of total? average?

  Prevention: Read the full question carefully, identify the exact
              formula type needed, then locate only the cells
              required for THAT specific formula.
```

## 2. TYPES OF TABLE CHART QUESTIONS

---

### Type 1 — Direct Value Reading

```
"What was the profit of Company B in the year 2020?"
→ Locate row: Year 2020.
→ Locate column: Company B — Profit sub-column.
→ Read the cell value. Done.

Trap: In multi-variable (nested) tables, "Company B" may have two sub-columns
      (Revenue and Cost). Reading Revenue instead of Cost = wrong cell.
      Always confirm you are in the correct sub-column before reading.
```

---

### Type 2 — Row Total or Column Total

```
"What is the total sales of all companies in 2021?"
→ Sum ALL cells in the 2021 row (across all company columns).

"What is the total sales of Company A over all given years?"
→ Sum ALL cells in the Company A column (across all year rows).

Key: Row direction = left-right (across). Column direction = top-bottom (down).
```

---

### Type 3 — Difference Between Two Cells

```
"What is the difference between Revenue of Company A and Company C in 2019?"
→ Read both cells from the SAME ROW (2019) → subtract.

"By how much did Company A's sales increase from 2018 to 2022?"
→ Read cells from TWO DIFFERENT ROWS in Company A's column → subtract.
```

---

### Type 4 — Ratio Between Two Cells

```
"Find the ratio of exports in 2017 to exports in 2021."
→ Read both values from the Exports column → simplify by HCF → ratio.
```

---

### Type 5 — Average of a Row or Column

```
"What is the average annual revenue of Company B?"
→ Sum all values in Company B's COLUMN → divide by number of years.

"What is the average sales across all companies in 2021?"
→ Sum all values in the 2021 ROW → divide by number of companies.
```

---

### Type 6 — Percentage Change

```
"By what % did production increase from 2018 to 2022?"
→ Old = 2018 value (base / denominator).
→ New = 2022 value.
→ % Change = [(New − Old) / Old] × 100.

Base = ALWAYS the earlier (Old) value. No exceptions.
```

---

### Type 7 — Percentage Share

```
"Company C's revenue in 2021 is what % of total revenue of all companies in 2021?"
→ Numerator = Company C's 2021 value.
→ Denominator = Sum of ALL companies' values in 2021 (row total).
→ % = (C's value / Row total) × 100.
```

---

### Type 8 — Derived Value

```
"In which year was profit (= Revenue − Cost) maximum?"
→ For each row: compute Profit = Revenue cell − Cost cell.
→ Identify the row with the highest resulting profit.

Profit is a DERIVED value — not directly in the table, but computable from two given columns.
```

---

### Type 9 — Maximum / Minimum

```
"Which company had the highest average sales over all years?"
→ Compute column total for each company.
→ When all columns have the SAME number of rows: Max total = Max average.
→ No need to divide — just compare totals.
```

---

### Type 10 — Missing Value Computation

```
"The total revenue of Company A over 5 years is ₹500 crores.
 Values for 4 years are given. Find the missing year's value."
→ Missing value = Total − Sum of all given values.

This is the most direct application of: Missing cell = Total − Others.
```

```
TABLE CHART DATA INTERPRETATION
        │

        │       ├── Multi-variable table (3+ variable columns)

        │       ├── Cumulative / running total table

        │       ├── Column headers (what each column measures)
        │       ├── Units (₹ Lakhs? Crores? Thousands? % ? Units?)
        │       ├── Grand total row/column (bottom row or right column)
        │       └── Footnotes (special conditions on specific cells)
        │
        ├── CALCULATION TYPES
        │       ├── Row total (sum across all columns for one row)
        │       ├── Column total (sum down all rows for one column)
        │       ├── Grand total (sum of everything)
        │       ├── Difference between cells
        │       ├── Ratio (A : B, simplified by HCF)
        │       ├── Average (sum ÷ count, for any row or column)
        │       ├── % change between rows (over time)
        │       ├── % share (cell / total × 100)
        │       ├── % more / less than
        │       └── Derived values (profit, savings, trade balance)
        │
        ├── ANALYSIS DIRECTIONS
        │       ├── Row-wise (compare all categories for ONE time period)
        │       ├── Column-wise (track ONE entity across all time periods)
        │       └── Diagonal (rare — patterns across row+column combinations)
        │
        └── COMMON TRAPS
                ├── Reading wrong cell (right row, wrong column)
                ├── Using wrong base in % change (new instead of old)
                ├── Summing wrong set of cells for totals
                ├── Confusing row average with column average
                ├── Forgetting to convert units
                ├── Confusing "% more than" with "% of"
                ├── Applying % from different rows/totals without computing actual values
                └── Missing the "given total" clue in missing-data questions
```

## 4. CONCEPT 1 — READING A TABLE CHART

> Unlike pie charts (visual slices) and bar graphs (visual heights), table charts are pure text and numbers in a grid. **Accuracy of navigation — finding the right cell — is the primary skill.** Speed comes from reading quickly; accuracy comes from always confirming both row AND column before reading.

---

### 4.1 The Five Reading Layers

**Read all five layers completely before touching any question. This 30-second investment prevents errors across every question in the set.**

---

#### LAYER 1 — Title

```
The title tells you what the ENTIRE TABLE represents.

Examples:
→ "Sales (in ₹ Lakhs) of Five Companies from 2017 to 2022"
   Meaning: All cells = Sales values | Unit = ₹ Lakhs | 5 companies × 6 years

→ "Marks obtained by Students in 5 Subjects (Max marks per subject = 100)"
   Meaning: All cells = Marks scored | Rows = Students | Columns = Subjects

→ "Import and Export Data (in ₹ Billion) by Country — Year 2022"
   Meaning: Each country has two sub-columns: Imports and Exports

What the title tells you instantly:
✓ What the rows represent (years? companies? students? cities?)
✓ What the columns represent (variables measured)
✓ What unit applies to all cells
✓ The scope of data (which years, which entities)
```

---

#### LAYER 2 — Column Headers

```
Column headers define WHAT IS BEING MEASURED in each column.

Simple table: One variable per column.
  | Year | Company A | Company B | Company C |

Nested/compound table: One main column has sub-columns.
        |    Company A    |    Company B    |
  | Year | Revenue | Cost  | Revenue | Cost  |

CRITICAL RULE FOR NESTED TABLES:
Before reading any cell in a nested table, confirm TWO things:
1. Which MAIN column are you in? (Company A? Company B?)
2. Which SUB-COLUMN are you in? (Revenue? Cost?)

Reading Company B's Revenue instead of Company A's Revenue
is a silent error — the number exists in the table, it just
belongs to a different entity.
```

---

#### LAYER 3 — Row Headers

```
Row headers define WHICH ENTITY the data in that row belongs to.

Common row types:
→ Years:     2017, 2018, 2019, 2020, 2021, 2022
→ Companies: Company A, Company B, Company C, Company D
→ Countries: India, China, USA, UK, Japan
→ Students:  Student 1, Student 2, Student 3
→ Cities:    Delhi, Mumbai, Bengaluru, Chennai

COUNT the rows before starting. Know how many entities are present.
If there are 6 year-rows, column sums have 6 addends.
If the question says "over 5 years" but the table has 6 years — check carefully.
```

---

#### LAYER 4 — Units

```
The unit is usually stated in:
→ The title: "Revenue (in ₹ Crores)"
→ A note below: "All values in thousands"
→ The column header: "Production (lakh units)"

CRITICAL UNIT RULE:
Do ALL calculations in the UNIT given in the table.
Convert to the required unit ONLY at the very final step.

Example:
Table unit = ₹ Lakhs.
Question asks for the answer in ₹ Crores.

WRONG approach: Convert each cell to Crores first, then add.
RIGHT approach: Add all cells (in Lakhs) → then divide the SUM by 100 → answer in Crores.

Why? Converting each cell introduces rounding errors at each step.
     Converting the final sum is ONE operation instead of many.

Common conversion factors:
  Thousands  → Lakhs  : ÷ 100
  Lakhs      → Crores : ÷ 100
  Crores     → Lakhs  : × 100
  Millions   → Crores : × 10  (1 Crore = 10 Million)
  Billions   → Crores : × 100 (1 Billion = 100 Crores)
```

---

#### LAYER 5 — Totals, Grand Totals, and Special Rows

```
Many tables include:
→ A "Total" row at the bottom: sum of all values in each column.
→ A "Total" column on the right: sum of all values in each row.
→ A "Grand Total" cell (bottom-right): sum of everything.

If a Total row/column IS given:
→ Verify your computation against it.
→ Use it to find missing cells: Missing = Total − (Sum of all other known cells).

If NO Total is given:
→ Compute it yourself for questions that need it.
→ Cross-verify: Sum of row totals = Sum of column totals = Grand total.

Special rows/columns sometimes present:
→ "Average" row: pre-computed average of each column.
→ "Growth %" column: pre-computed % change from previous row.
→ These save time — use them when present, compute them when absent.
```

---

### 4.2 Rows vs Columns — Never Confuse These

```
THE MOST FUNDAMENTAL RULE IN TABLE DI:

ROW    = horizontal direction (left → right)
         = one entity's data across ALL variables
         = reading across one single horizontal band

COLUMN = vertical direction (top → bottom)
         = one variable's data across ALL entities / time periods
         = reading down one single vertical band

─────────────────────────────────────────────────────────
Visual Layout:
         | Col 1 | Col 2 | Col 3 | ← COLUMN HEADERS
Row 1    |  a₁₁  |  a₁₂  |  a₁₃  | ← Row 1: a₁₁ + a₁₂ + a₁₃ = Row 1 total
Row 2    |  a₂₁  |  a₂₂  |  a₂₃  | ← Row 2 total
Row 3    |  a₃₁  |  a₃₂  |  a₃₃  | ← Row 3 total
           ↑       ↑       ↑
         Col 1   Col 2   Col 3    ← each column sums downward
         total   total   total
─────────────────────────────────────────────────────────

QUESTION → DIRECTION:

"Total sales of ALL companies in 2021"
→ 2021 is a ROW → SUM ACROSS that row (left to right).

"Total sales of Company A over ALL years"
→ Company A is a COLUMN → SUM DOWN that column (top to bottom).

"Average sales across all companies in 2021"
→ Sum the 2021 ROW → divide by number of companies.

"Average annual sales of Company B"
→ Sum Company B's COLUMN → divide by number of years.

Getting the direction wrong = completely wrong calculation every single time.
```

---

### 4.3 Missing Values and Derived Columns

```
MISSING VALUE — TYPE 1: One cell in a row is blank, row total is given.
→ Missing cell = Row total − Sum of all other known cells in that row.

MISSING VALUE — TYPE 2: One cell in a column is blank, column total is given.
→ Missing cell = Column total − Sum of all other known cells in that column.

MISSING VALUE — TYPE 3: A % or ratio clue is given.
→ "Company B's 2021 sales are 20% more than its 2020 sales."
   If 2020 value = 200: 2021 value = 200 × 1.20 = 240.

→ "Ratio of 2022 to 2021 production is 4:3."
   If 2021 = 300: 2022 = (4/3) × 300 = 400.

MISSING VALUE — TYPE 4: Average is given.
→ "Average of 5 years' data = 250."
   Total = 250 × 5 = 1250.
   Missing = 1250 − (Sum of 4 given values).

DERIVED COLUMN: A column NOT in the table but computable from existing columns.
→ Profit = Revenue − Cost (Revenue and Cost columns are given; Profit is derived).
→ Trade Balance = Exports − Imports.
→ % Margin = (Profit / Revenue) × 100.

For any derived-column question:
Step 1: Compute the derived value for EACH required row.
Step 2: Apply the formula asked (max? ratio? % change?) to these computed values.
```

---

### 4.4 Reading Multi-Variable Nested Tables

```
Nested tables display multiple variables for multiple entities simultaneously.

Example layout:
           |      Company A      |      Company B      |
| Year     | Revenue |   Cost    | Revenue |   Cost    |
|----------|---------|-----------|---------|-----------|
| 2018     |   500   |    420    |   400   |    350    |
| 2019     |   620   |    510    |   480   |    400    |
| 2020     |   480   |    490    |   520   |    430    |

Reading "Company A's Profit in 2019":
Step 1: Go to 2019 row.
Step 2: Move to Company A's section (left half).
Step 3: Read Revenue = 620, Cost = 510.
Step 4: Compute: 620 − 510 = 110.

THE DANGER ZONE:
After reading Company A's Revenue (620), your eyes naturally move right
to Company B's Revenue (480) — but the COST you need is 510, which is
to the RIGHT of 620 under Company A, NOT the 480 under Company B.

Prevention protocol:
1. Read the first value (620) and note it: "A_Rev_2019 = 620."
2. Move WITHIN Company A's section to find Cost: "A_Cost_2019 = 510."
3. Only then compute: 620 − 510 = 110.
Never look at two values simultaneously — read one, record it, then read the next.
```

## 5. CONCEPT 2 — TYPES OF TABLE CHARTS

---

### 5.1 Simple Data Table

One variable per column. The most basic and most common format.

```
Example: Annual Sales of Companies (in ₹ Crores)

| Year | Company A | Company B | Company C | Company D |
|------|-----------|-----------|-----------|-----------|
| 2018 |    200    |    150    |    180    |    120    |
| 2019 |    240    |    180    |    210    |    150    |
| 2020 |    180    |    200    |    190    |    160    |
| 2021 |    300    |    250    |    260    |    200    |
| 2022 |    360    |    300    |    290    |    240    |

Structure: Years as rows, Companies as columns.
Each cell = that company's sales in that year.

Navigation examples:
"Company B's sales in 2020" → Row: 2020, Column: Company B → 200.
"Total sales of Company A" → Sum A's column: 200+240+180+300+360 = 1280.
"Total sales across all companies in 2021" → Sum 2021 row: 300+250+260+200 = 1010.
```

---

### 5.2 Comparative Two-Variable Table

Two closely related variables appear side by side, enabling direct comparison and derivation.

```
Example: Revenue and Expenditure (in ₹ Lakhs)

| Year | Revenue | Expenditure |
|------|---------|-------------|
| 2018 |   500   |    420      |
| 2019 |   620   |    510      |
| 2020 |   480   |    490      |
| 2021 |   750   |    600      |
| 2022 |   900   |    680      |

Derived column (not in table, but computable):
  Profit = Revenue − Expenditure
  2018:  80 | 2019: 110 | 2020: −10 (LOSS) | 2021: 150 | 2022: 220

Questions this format generates:
→ "Which year was a loss year?" → 2020 (Cost > Revenue).
→ "Which year had maximum profit?" → 2022 (₹220 Lakhs).
→ "In which year was % profit margin maximum?" → (Profit/Revenue × 100 for each year).
→ "By what % did Revenue grow from 2019 to 2022?" → [(900−620)/620] × 100.
```

---

### 5.3 Multi-Variable Table (3+ Columns)

Multiple variables for multiple entities simultaneously. Most common format in competitive exams.

```
Example: Marks scored by Students (Max = 100 per subject, Max total = 500)

| Student | Maths | Science | English | History | Geography | Total |  %   |
|---------|-------|---------|---------|---------|-----------|-------|------|
| A       |  85   |   78    |   92    |   70    |    75     |  400  | 80%  |
| B       |  72   |   88    |   76    |   65    |    80     |  381  |76.2% |
| C       |  91   |   95    |   88    |   85    |    82     |  441  |88.2% |
| D       |  65   |   70    |   82    |   60    |    68     |  345  | 69%  |
| E       |  80   |   75    |   79    |   72    |    78     |  384  |76.8% |

Notes:
→ "Total" column = derived (sum of 5 subject scores).
→ "%" column = derived (Total / 500 × 100).
→ Questions can ask about individual subjects, totals, %, or comparisons.

Reading rule: Always specify row (student) AND column (subject) before reading.
"Student C's Science score" → Row: C, Column: Science → 95. Not 88 (English) or 82 (Geography).
```

---

### 5.4 Percentage Table

Cells contain PERCENTAGE values instead of absolute values. Requires special handling.

```
Example: Market share (%) of Products by Region
(Each row sums to 100% — showing product-wise distribution within each region)

| Region  | Product A | Product B | Product C | Product D |
|---------|-----------|-----------|-----------|-----------|
| North   |    35%    |    25%    |    20%    |    20%    |
| South   |    20%    |    40%    |    25%    |    15%    |
| East    |    30%    |    20%    |    30%    |    20%    |
| West    |    25%    |    30%    |    15%    |    30%    |

CRITICAL UNDERSTANDING — TWO STRUCTURES:

Structure 1 — Row sums to 100%:
→ Each row shows distribution of one region across products.
→ To get actual values: need each ROW's total (total sales of that region).

Structure 2 — Column sums to 100%:
→ Each column shows distribution of one product across regions.
→ To get actual values: need each COLUMN's total (total sales of that product).

If North region's total sales = ₹800 Crores:
Product A in North = 35% × 800 = ₹280 Crores.
Product B in North = 25% × 800 = ₹200 Crores.

Ratio of A to B within North (same total):
Use % directly: 35:25 = 7:5. (Same total cancels.)

Ratio of A (North) to A (South) — DIFFERENT totals:
MUST compute actual values: 280 vs (20% × South total).
CANNOT use 35%:20% directly.
```

---

### 5.5 Cumulative / Running Total Table

Values accumulate over time. Each cell includes all previous values.

```
Example: Cumulative Production (in lakh units)

| Quarter | Production (Current Qtr) | Cumulative (Year-to-Date) |
|---------|--------------------------|---------------------------|
|   Q1    |           30             |            30             |
|   Q2    |           40             |            70             |
|   Q3    |           25             |            95             |
|   Q4    |           45             |           140             |

Key formulas for cumulative tables:
→ Individual period value = Cumulative at period N − Cumulative at period N−1.
   Q3 individual = 95 − 70 = 25 ✓
→ Total annual production = Final cumulative value = 140.
→ Best quarter = highest individual value (Q4 = 45).

The cumulative column NEVER decreases (unless negative values exist).
If you see the cumulative value fall → you've misread a cell.
```

---

### 5.6 Table with Missing Data

One or more cells are intentionally blank. A given condition allows computation.

```
Example: Sales Data (in ₹ Crores)

| Year | Company A | Company B | Company C | Row Total |
|------|-----------|-----------|-----------|-----------|
| 2018 |    200    |    150    |    180    |    530    |
| 2019 |    240    |     ?     |    210    |    630    |
| 2020 |    180    |    200    |     ?     |    570    |
| 2021 |     ?     |    250    |    260    |    810    |

Missing value computation:
2019, Company B: ? = 630 − 240 − 210 = 180.
2020, Company C: ? = 570 − 180 − 200 = 190.
2021, Company A: ? = 810 − 250 − 260 = 300.

ALWAYS compute missing values BEFORE reading the questions.
The missing cell is often the very thing being asked about!
Spending 20 seconds here saves 2 minutes of confusion later.
```

---

### 5.7 Two-Way / Matrix Table

Both rows AND columns represent categories. Each cell = value for that specific (row, column) combination.

```
Example: Number of Employees by City and Department

|           | Sales | IT  | HR  | Finance | Ops | Total |
|-----------|-------|-----|-----|---------|-----|-------|
| Delhi     |  150  | 200 | 80  |   120   | 250 |  800  |
| Mumbai    |  200  | 250 | 100 |   150   | 300 | 1000  |
| Bengaluru |  100  | 300 | 60  |    90   | 200 |  750  |
| Chennai   |  120  | 180 | 70  |   110   | 220 |  700  |
| Total     |  570  | 930 | 310 |   470   | 970 | 3250  |

Reading specific cells:
"IT employees in Mumbai" → Row: Mumbai, Col: IT → 250.
"Total HR employees" → HR column total → 310.
"Total employees in Bengaluru" → Bengaluru row total → 750.
"Grand total employees" → Bottom-right cell → 3250.

Key calculations unique to matrix tables:
"IT employees in Delhi as % of total IT employees"
= (200/930) × 100 = 21.51%

"Sales employees in Mumbai as % of total Mumbai employees"
= (200/1000) × 100 = 20%

"Which city has the highest % of IT employees within its total?"
→ Delhi: 200/800=25% | Mumbai: 250/1000=25% | Bengaluru: 300/750=40% ← Max
→ Answer: Bengaluru.
```

## 6. CONCEPT 3 — CORE CALCULATIONS

> Every table chart question uses one or more of these eight calculation types. Master each formula: what it is, when to use it, what the base is, and what the common trap is.

---

### 6.1 Row Totals and Column Totals

```
ROW TOTAL = Sum of all cells in one horizontal row.
→ Represents: One entity's data aggregated across all variables.
→ Example: "Total sales of ALL companies in 2021" = Sum of the 2021 row.

COLUMN TOTAL = Sum of all cells in one vertical column.
→ Represents: One variable aggregated across all entities / time periods.
→ Example: "Total sales of Company A over all years" = Sum of Company A's column.

GRAND TOTAL = Sum of ALL cells in the entire table.
→ Cross-verify: Grand total = Sum of ALL row totals = Sum of ALL column totals.
→ Mismatch = arithmetic error somewhere.

Quick addition technique — Pair complementary numbers:
Sum: 200 + 240 + 180 + 300 + 360
Pair: (200+360) + (240+300) + 180 = 560 + 540 + 180 = 1280.
Much faster than sequential addition.
```

---

### 6.2 Difference Between Cells

```
Formula: Difference = Larger value − Smaller value

SAME ROW (comparing two variables for the same entity/year):
Example: "How much more did Company A earn than Company B in 2021?"
→ A_2021 − B_2021.

SAME COLUMN (comparing the same variable across two time periods):
Example: "How much did Company A's sales increase from 2019 to 2022?"
→ A_2022 − A_2019.

DIFFERENT ROW AND COLUMN (any two cells):
→ Just read both cells correctly and subtract.
→ Example: "Difference between Company A in 2018 and Company C in 2021."
→ Read A_2018 and C_2021. Subtract smaller from larger.

KEY QUESTION TO ASK: Does the question want an absolute difference (units)
or a percentage difference (%)? Never compute one when the other is asked.
```

---

### 6.3 Ratio Between Cells

```
Formula: Ratio = Value₁ : Value₂
         Simplify by dividing both by their HCF.

Fast HCF approach:
Step 1: If both end in 0, cancel trailing zeros first.
  1200 : 1800 → 12 : 18 (cancel one 0 each)
Step 2: Divide by obvious common factors: 2, 3, 5, 7...
  12 : 18 → 6 : 9 → 2 : 3.
Step 3: Stop when no common factor remains.

Quick examples:
360 : 480 → 36:48 → 9:12 → 3:4
750 : 600 → 75:60 → 15:12 → 5:4
240 : 160 → 24:16 → 3:2
1050 : 700 → 105:70 → 15:10 → 3:2

Ratios can be between ANY two cells (same row, same column, or different).
No constraint on which cells can be compared in a ratio question.
```

---

### 6.4 Average of a Row or Column

```
Formula: Average = Sum ÷ Count

COLUMN AVERAGE (one entity over all years):
"Average annual revenue of Company B"
→ Sum all cells in Company B's column ÷ number of year-rows.

ROW AVERAGE (all entities in one year):
"Average sales across all companies in 2021"
→ Sum all cells in the 2021 row ÷ number of companies.

MAX AVERAGE SHORTCUT (when all columns have the SAME number of rows):
→ Maximum average column ↔ Maximum total column.
→ Compare totals instead of computing averages. Save one division per column.

DEVIATION METHOD (faster when values are close together):
1. Pick a convenient reference R near the middle.
2. Deviations: dᵢ = valueᵢ − R (positive or negative).
3. Average = R + (Sum of all dᵢ) ÷ Count.

Example: Values = 200, 240, 180, 300, 360. R = 240.
Deviations: −40, 0, −60, +60, +120. Sum = +80.
Average = 240 + (80/5) = 240 + 16 = 256.

Check: (200+240+180+300+360)/5 = 1280/5 = 256. ✓
```

---

### 6.5 Percentage Change Between Rows

```
Formula: % Change = [(New − Old) / Old] × 100
         BASE = ALWAYS the Old (earlier / starting) value.

Positive result → increase. Negative result → decrease.

BETWEEN CONSECUTIVE ROWS (year-over-year):
"By what % did Company A's sales increase from 2020 to 2021?"
→ Old = 2020 value. New = 2021 value. Base = 2020.

BETWEEN NON-CONSECUTIVE ROWS:
"By what % did Company A's sales change from 2018 to 2022?"
→ Old = 2018 value. New = 2022 value. Base = 2018.

PHRASING TRAPS — ALL MEAN OLD = EARLIER YEAR:
"Growth FROM 2018 TO 2022"       → Old = 2018. ✓
"Change BETWEEN 2017 AND 2022"   → Old = 2017. ✓
"Change OVER the period 2019–22" → Old = 2019. ✓
"Growth in 2022 COMPARED TO 2020"→ Old = 2020. ✓

THE ONE WRONG VERSION:
"% change FROM 2022 TO 2018" — chronologically 2018 is earlier.
Even though question says "from 2022", Old = 2018 (base).
→ This trips up students. Always: earlier year = Old = Base. Always.
```

---

### 6.6 Percentage Share of a Cell

```
Formula: % Share = (Cell Value / Total) × 100

The "total" depends on WHAT the question asks:

1. % share of ROW total:
   "Company C's 2021 sales as % of all companies' sales in 2021"
   → Denominator = 2021 row total.

2. % share of COLUMN total:
   "Company C's 2021 sales as % of Company C's total over all years"
   → Denominator = Company C's column total.

3. % share of GRAND total:
   "Company C's 2021 sales as % of all sales in all years"
   → Denominator = grand total of entire table.

EXAM TRAP: "Company C's 2021 sales as % of [what]?"
The word "[what]" tells you the denominator. Read carefully.

FRIENDLY TOTAL SHORTCUTS (avoid division):
Total = 500  → % of V = V × (100/500) = V/5
Total = 1000 → % of V = V/10
Total = 200  → % of V = V/2
Total = 2000 → % of V = V/20
Total = 400  → % of V = V/4
```

---

### 6.7 What % more / less — Between Cells

```
THREE DIFFERENT FORMULAS — NEVER MIX THEM:

Formula 1: "A is what % OF B?"
→ (A / B) × 100
→ Measures A as a proportion of B (includes the base itself).
→ A = 300, B = 200: (300/200)×100 = 150% of B.

Formula 2: "A is what % MORE THAN B?"
→ [(A − B) / B] × 100
→ Measures only the EXCESS of A over B, as % of B.
→ A = 300, B = 200: [(300−200)/200]×100 = 50% more than B.

Formula 3: "A is what % LESS THAN B?"
→ [(B − A) / B] × 100
→ Measures only the SHORTFALL of A below B, as % of B.
→ A = 150, B = 200: [(200−150)/200]×100 = 25% less than B.

THE GOLDEN RULE:
Whatever comes after the word "THAN" = the BASE (denominator).
Whatever comes after the word "OF" = the BASE (denominator).

RELATIONSHIP BETWEEN THE THREE:
"% of B" = 100 + "% more than B" → 150% = 100 + 50% ✓
"% of A" = 100 − "% less than A" → 75% = 100 − 25% ✓

Use this relationship to instantly verify your answer.
```

---

### 6.8 Derived Values — Profit, Net, Balance

```
A DERIVED VALUE is NOT directly in the table.
It is computed from two or more given columns for each row.

Common derived values in table DI:

  Profit / Loss     = Revenue − Cost (or Sales − Expenditure)
  Net Savings       = Income − Expenditure
  Trade Balance     = Exports − Imports (positive = surplus; negative = deficit)
  % Profit Margin   = (Profit / Revenue) × 100
  Per Capita        = Total Value / Population

COMPLETE STRATEGY for derived value questions:

Step 1: Identify which columns combine to give the derived value.
Step 2: Compute the derived value for EVERY relevant row.
        (Don't compute only for the year being asked about — you might need to compare.)
Step 3: Apply the formula the question asks for (max? ratio? % change?) to derived values.

Full Example:
| Year | Revenue | Cost |  Profit (derived) | % Margin (derived)  |
|------|---------|------|-------------------|---------------------|
| 2018 |   500   |  420 |        80         |    80/500=16%       |
| 2019 |   620   |  510 |       110         |   110/620=17.74%    |
| 2020 |   480   |  490 |       −10 (LOSS)  |  −10/480=−2.08%     |
| 2021 |   750   |  600 |       150         |   150/750=20%       |
| 2022 |   900   |  680 |       220         |   220/900=24.44%    |

"In which year was % profit margin maximum?" → 2022 (24.44%).
"In which year did the company operate at a loss?" → 2020.
"What is the average annual profit?" → (80+110−10+150+220)/5 = 550/5 = ₹110 Lakhs.
```

## 7. CONCEPT 4 — ROW VS COLUMN ANALYSIS

---

### 7.1 Row-wise Analysis

```
Row-wise analysis = examine ONE row, compare values across columns.

WHEN TO USE IT:
→ Questions about ONE specific year/entity across multiple variables.
→ "Which company had the highest sales IN 2021?" → scan 2021 row.
→ "In 2020, what is the ratio of Company A to Company C?" → same row.
→ "Total sales of all companies in 2019" → sum the 2019 row.

HOW TO DO IT:
1. Locate the relevant row by its row header.
2. Read values from all required columns in THAT row.
3. Compute (sum, ratio, max, %).

Example:
| Year | A  | B  | C  | D  |
|------|----|----|----|----|
| 2021 | 80 | 60 | 90 | 70 |

Row 2021: C(90) > A(80) > D(70) > B(60).
Max in 2021 = C. Row total = 300. A as % of row total = 80/300×100 = 26.67%.
```

---

### 7.2 Column-wise Analysis

```
Column-wise analysis = examine ONE column, compare values across rows.

WHEN TO USE IT:
→ Questions about ONE entity/variable across multiple years.
→ "In which year was Company A's sales highest?" → scan Company A's column.
→ "By what % did Company B's sales change from 2018 to 2022?" → Company B's column.
→ "Average annual sales of Company C" → sum column C, divide by years.

HOW TO DO IT:
1. Locate the relevant column by its column header.
2. Read values from all required rows in THAT column.
3. Compute (sum, ratio, max, average, % change).

Example:
| Year | Company A |
|------|-----------|
| 2018 |    200    |
| 2019 |    240    |
| 2020 |    180    |
| 2021 |    300    |
| 2022 |    360    |

Column A: Highest = 360 (2022). Lowest = 180 (2020).
Average = 1280/5 = 256. % change 2018→2022 = [(360−200)/200]×100 = 80%.
```

---

### 7.3 Finding Maximum and Minimum

```
MAXIMUM / MINIMUM IN A ROW:
→ Scan that row left to right, identify the largest / smallest cell.

MAXIMUM / MINIMUM IN A COLUMN:
→ Scan that column top to bottom, identify the largest / smallest cell.

MAXIMUM / MINIMUM IN THE ENTIRE TABLE:
→ Compare all cells — usually visible by scanning and finding the extreme value.

FOR AVERAGES:
→ Max average column = Max total column (when all columns have same row count).
→ Compare TOTALS instead of computing averages. Saves n divisions.

FOR DERIVED VALUES (like profit):
→ Max profit ≠ Max revenue AND ≠ Min cost independently.
→ Must compute profit for EACH row, then find the max derived value.

VISUAL SHORTCUT:
→ Very large differences (one value clearly dominates): read directly.
→ Close values: compute exactly.
→ Never guess when two values might be close — compute both.
```

## 8. CONCEPT 5 — PERCENTAGE TABLES

> Percentage tables require extra care because the cells are already percentages. You must understand what denominator (total) each % refers to before doing any calculation.

---

### Types of Percentage Table Structure

```
STRUCTURE 1 — Row % (each ROW sums to 100%):
→ Shows how each row's total is distributed across columns.
→ To get actual values: need each ROW'S total.
→ Actual value = Cell% × Row total ÷ 100.

STRUCTURE 2 — Column % (each COLUMN sums to 100%):
→ Shows how each column's total is distributed across rows.
→ To get actual values: need each COLUMN'S total.
→ Actual value = Cell% × Column total ÷ 100.

STRUCTURE 3 — % of Grand Total (all cells together ≈ 100%):
→ Each cell shows its share of the total across EVERYTHING.
→ To get actual value: need the grand total.
→ Actual value = Cell% × Grand total ÷ 100.
```

---

### Key Rules for Percentage Tables

```
RULE 1 — Within same row (same total) → use % directly for ratios:
  Row: North (total = ₹800 Cr). A=35%, B=25%.
  Ratio A:B = 35:25 = 7:5. (Same total cancels out.)
  "A is what % more than B?" = [(35−25)/25]×100 = 40%. (Same total → same result.)

RULE 2 — Across different rows (different totals) → must compute actual values:
  North (total=800): A=35% → A_North = 280.
  South (total=600): A=20% → A_South = 120.
  "A_North is what % more than A_South?"
  WRONG: [(35−20)/20]×100 = 75%.
  RIGHT: [(280−120)/120]×100 = 133.33%.
  Different totals → can NEVER compare % directly.

RULE 3 — Total of a product/variable across all rows:
  Total of Product A = Sum of (A's % × each row's total) across all rows.
  = (35%×800) + (20%×600) + (30%×400) + (25%×500)
  = 280 + 120 + 120 + 125 = 645.
```

## 9. CONCEPT 6 — TWO-WAY / MATRIX TABLES

---

### Reading Protocol for Matrix Tables

```
In a two-way table, EVERY cell requires TWO identifiers:
→ Row label (e.g., city name)
→ Column label (e.g., department name)

Before reading any cell, state BOTH: "I need row [X], column [Y]."

Example protocol:
"IT employees in Mumbai" → Row: Mumbai, Column: IT → 250.
"HR employees in Delhi" → Row: Delhi, Column: HR → 80.
"NOT: IT in Mumbai = HR in Delhi" (they come from different rows+columns).

EDGE CASE: Row and column totals in the same table.
"Total IT employees" → IT COLUMN total → sum down IT column.
"Total Mumbai employees" → Mumbai ROW total → sum across Mumbai row.
"Total employees overall" → Grand total = Sum of row totals = Sum of col totals.
```

---

### Percentage Questions Unique to Matrix Tables

```
TYPE 1: "Cell as % of its row total"
"Sales employees in Mumbai as % of total Mumbai employees."
= (Mumbai Sales / Mumbai total) × 100 = (200/1000) × 100 = 20%.

TYPE 2: "Cell as % of its column total"
"IT employees in Bengaluru as % of total IT employees."
= (Bengaluru IT / IT column total) × 100 = (300/930) × 100 = 32.26%.

TYPE 3: "Cell as % of grand total"
"HR employees in Delhi as % of all employees."
= (Delhi HR / Grand total) × 100 = (80/3250) × 100 = 2.46%.

TYPE 4: "Which row has highest % of a given column?"
"Which city has the highest % of IT employees within its total?"
Compute IT% for each city: Delhi=200/800=25%, Mumbai=250/1000=25%,
Bengaluru=300/750=40% ← Max, Chennai=180/700=25.7%.
Answer: Bengaluru.
```

## 10. CONCEPT 7 — TABLES WITH MISSING DATA

---

### Complete Framework for Finding Missing Values

```
STEP 1: Identify what type of clue is given.
  → Row total given? → Missing cell = Row total − known cells in that row.
  → Column total given? → Missing cell = Col total − known cells in that column.
  → Average given? → Total = Average × Count → Missing = Total − known sum.
  → % clue given? → Compute from the known value + the % relationship.
  → Ratio clue given? → Use the ratio to express the missing value, then solve.

STEP 2: Compute ALL missing values before reading any question.

STEP 3: Double-check: does your computed missing value make the row/column
        total consistent? If not, recompute.

EXAMPLES of each clue type:

Row total: Missing = 630 − 240 − 210 = 180. Done.

Average: "Average over 5 years = 272. Find the missing year."
Total = 272×5 = 1360. Missing = 1360−(200+240+300+360) = 260.

% clue: "2021 value is 25% more than 2020 value (= 240)."
2021 = 240 × 1.25 = 300.

Ratio clue: "2022:2021 production = 4:3. 2021 = 300."
2022 = (4/3) × 300 = 400.
```

## 11. CONCEPT 8 — COMBINING TABLE WITH OTHER CHARTS

---

### Table + Bar Graph

```
Typical setup:
→ Bar graph gives total revenue / production by year (the big picture).
→ Table gives the breakdown by category / product within each year.

HOW TO COMBINE:
→ From bar graph: Read total for Year Y.
→ From table: Read category X's % or absolute value.
→ If table has % → Category value = Table% × Bar total ÷ 100.
→ If table has absolute values → use directly.

Common question: "Find the total using the bar + table combination."
→ Bar gives one category's value. Table gives that category's %.
→ Total = (bar value) / (table %) × 100.
```

---

### Table + Pie Chart

```
Typical setup:
→ Pie chart shows % distribution of total across departments.
→ Table shows department-wise sub-breakdown (e.g., gender split).

TWO-STEP PROTOCOL (identical to pie + table from previous guide):
Step 1 (Pie): Department A = X% of total T → A's count = X% × T.
Step 2 (Table): Gender ratio in A → Male in A = (Male%) × A's count.

NEVER skip directly to the table.
NEVER apply the table % to the grand total directly.
```

---

### Table + Line Graph

```
Typical setup:
→ Line graph shows overall index or growth rate over time.
→ Table shows actual values for a base year.

HOW TO COMBINE:
→ From table: Get base year value.
→ From line: Get growth rate for subsequent years.
→ Projection: Apply growth rate to base value.

Example:
Base year (from table) = 500 units.
Growth rate Year 1 (from line) = 20%.
Year 1 projected = 500 × 1.20 = 600 units.
```

## 12. MASTER TRICKS & SHORTCUTS

---

### 🚀 TRICK 1 — The Cell Address System (Eliminates Wrong Cell Reads)

```
Before reading ANY cell, mentally state BOTH coordinates:
"Row = [entity], Column = [variable]."

Then navigate: go to that ROW → stay in that row → find that COLUMN → read.

In nested tables, state THREE coordinates:
"Row = [year], Main Column = [company], Sub-Column = [revenue/cost]."

This habit prevents the #1 error in table DI:
reading the right row but the wrong column.

Example ritual:
Question: "Company B's profit in 2021."
Statement: "Row = 2021. Main col = Company B. Sub-col = Cost and Revenue."
Read: Revenue(B,2021) = X. Cost(B,2021) = Y. Profit = X−Y.

Three seconds. Zero wrong-cell errors.
```

---

### 🚀 TRICK 2 — Compute Missing Values Before Question 1

```
If ANY cell is blank or any total is given:
STOP. Compute the missing values FIRST.

Formula: Missing cell = Total − Sum of all other known cells in same row/column.

Why do this first?
→ The missing cell is often the subject of the first question.
→ Later questions may use the missing cell indirectly.
→ Not computing it upfront creates confusion mid-calculation.

Time investment: 20–30 seconds.
Time saved: 2–3 minutes of re-reading the table later.
```

---

### 🚀 TRICK 3 — % Change Base Is Always the Old (Earlier) Value

```
One rule. No exceptions. Old = Base = Denominator.

"Earlier in time" = "Old" = "Base."

% Change = [(New − Old) / Old] × 100.

Memory device: Think of "Old" as the foundation (base) of a building.
The "New" is built on top of that foundation.
The % change measures HOW MUCH it grew above/below the OLD foundation.

If someone says: "% change from 2022 TO 2018" — still: Old = 2018.
Time goes 2018 → 2022. Earlier year (2018) = Old = Base.
```

---

### 🚀 TRICK 4 — The "THAN" Rule for % More/Less

```
Whatever comes AFTER the word "THAN" = the BASE (denominator).

"A is what % more THAN B?"     → Base = B → [(A−B)/B] × 100.
"A is what % less THAN B?"     → Base = B → [(B−A)/B] × 100.
"A is what % more THAN 2020?"  → Base = 2020 value → [(A−2020)/2020] × 100.

INSTANT VERIFICATION using the relationship:
"% OF B" = 100 + "% MORE THAN B"

Example: A=360, B=300.
"A is what % more than B?" = (60/300)×100 = 20%.
"A is what % of B?" = (360/300)×100 = 120%.
Check: 120% = 100 + 20% ✓.

If your "% more than" and "% of" don't satisfy this relationship → recalculate.
```

---

### 🚀 TRICK 5 — Max Average = Max Total (Equal Row Counts)

```
When all columns have the SAME number of rows (which is almost always the case):
The column with the highest SUM has the highest AVERAGE.

You don't need to divide at all.
Just identify the column with the highest total.

Example:
Company A total = 1280 | Company B total = 1080 | Company C total = 1130.
"Which company has the highest average sales?"
→ A has the highest total → A has the highest average. Done.
No need to compute 1280/5, 1080/5, 1130/5.

EXCEPTION: This shortcut FAILS when columns have DIFFERENT row counts.
(Example: Company A has data for 6 years but Company B for only 5 years.)
In that case, compute averages explicitly.
```

---

### 🚀 TRICK 6 — Deviation Method for Fast Averages

```
When values in a row or column are close to each other:

1. Pick a convenient reference R near the middle of the values.
2. Compute deviation: dᵢ = valueᵢ − R (positive if above R, negative if below).
3. Average = R + (Sum of all deviations ÷ Count).

Example: Values = 240, 270, 250, 290, 260. R = 260.
Deviations: −20, +10, −10, +30, 0. Sum = +10.
Average = 260 + (10/5) = 260 + 2 = 262.

vs. Traditional: (240+270+250+290+260)/5 = 1310/5 = 262 ✓.

Saves time because you add small deviations (±20 range) instead of
adding large 3-digit numbers (200-series).
```

---

### 🚀 TRICK 7 — Pair Complementary Numbers for Fast Column Sums

```
When adding 4, 5, or 6 values, look for pairs that sum to round numbers.

Example: 200 + 240 + 180 + 300 + 360 + 220
Pair: (200+300=500) + (240+360=600) + (180+220=400).
Sum = 500 + 600 + 400 = 1500.

Example 2: 120 + 90 + 110 + 80 + 100
Pair: (120+80=200) + (90+110=200) + 100.
Sum = 200 + 200 + 100 = 500.

Look for pairs that sum to multiples of 100, 50, or 10 first.
The remaining odd values are easy to add once the pairs are done.
```

---

### 🚀 TRICK 8 — Fraction Comparison for Max % Change (No Division)

```
To find which period/entity had the highest % change:

1. For each period: compute Rise = New − Old (or Drop = Old − New).
2. Form fraction: Rise / Old (or Drop / Old).
3. Compare ALL fractions using cross-multiplication (no division needed).
4. Largest fraction = highest % change.

Cross-multiplication rule: Is a/b > c/d?
Compare a×d vs b×c. Larger product = larger fraction.

Example:
Period 1: Rise=40, Old=200 → fraction = 40/200.
Period 2: Rise=60, Old=240 → fraction = 60/240.
Compare: 40×240=9600 vs 60×200=12000. 12000>9600 → Period 2 is larger.

Period 3: Rise=30, Old=180 → fraction = 30/180.
Compare Period 2 vs 3: 60×180=10800 vs 30×240=7200. 10800>7200 → Period 2 still larger.

Max % change → Period 2 (2019→2020 or wherever it is in your table).
Zero divisions performed. All comparisons by multiplication.
```

---

### 🚀 TRICK 9 — Percentage Share with Friendly Totals

```
When the total is a "friendly" number, % share simplifies instantly:

Total = 500  → % of value V = V/500 × 100 = V/5.
Total = 1000 → % of V = V/10.
Total = 200  → % of V = V/2.
Total = 2000 → % of V = V/20.
Total = 400  → % of V = V/4.
Total = 2500 → % of V = V/25.

Example: V=350, Total=1000 → % = 350/10 = 35%. No long division needed.
Example: V=120, Total=400  → % = 120/4 = 30%. One step.

When total is NOT friendly → approximate for MCQ elimination:
(360/743)×100 ≈ (360/750)×100 = 48%. Options: 42%, 48%, 54% → Answer: 48%.
```

---

### 🚀 TRICK 10 — Grand Total Cross-Check

```
After computing row totals and column totals:
Sum of all row totals = Sum of all column totals = Grand total.

If these two sums DIFFER → you have an arithmetic error somewhere.
Find it before answering questions.

This verification takes 10 seconds and saves multiple wrong answers.

It also helps find the missing value when both a row total AND a column total
are given but one cell is missing:
Missing cell satisfies BOTH: row equation AND column equation.
Use either to solve (both give the same answer if no errors).
```

---

### 🚀 TRICK 11 — Reverse Calculation (Finding Old from % Change)

```
Exam question type: "Value in 2023 = ₹480 Cr. This is 20% more than 2022. Find 2022."

Formula: Old = New ÷ (1 + % change / 100)
= 480 ÷ 1.20 = 400.

TRAP — WRONG METHOD:
"20% of 480 = 96. 480 − 96 = 384."
This gives 384, NOT 400. WRONG.

WHY: The 20% is relative to 2022 (the Old value), NOT relative to 2023 (the New value).
Old × 1.20 = 480 → Old = 480/1.20 = 400. ✓

Similarly: "Value in 2023 = 360. This is 25% less than 2022. Find 2022."
Old = 360 ÷ (1 − 0.25) = 360 ÷ 0.75 = 480.

General formula: Old = New ÷ Multiplier.
(Increase by x% → Multiplier = 1 + x/100)
(Decrease by x% → Multiplier = 1 − x/100)
```

---

### 🚀 TRICK 12 — Approximation for MCQ Elimination

```
When options are spread apart (≥ 4 units difference), approximate:

Step 1: Round all values to nearest 5, 10, or 50.
Step 2: Compute with rounded values.
Step 3: Match to the nearest option.

Example: (236 / 718) × 100
Approximate: (240 / 720) × 100 = (240/720) × 100 = 33.33%.
Options: 28%, 33%, 38%, 43% → Answer: 33%.
Full calc: 236/718 × 100 = 32.87% ≈ 33% ✓.

Safe threshold: Use approximation when adjacent options differ by ≥ 4 units.
Exact calculation: Required when options are 32%, 33%, 34% (too close for approximation).
```

---

### 🚀 TRICK 13 — Percentage Table Ratio Rule

```
WITHIN the same row OR the same column (same underlying total):
Ratio of actual values = Ratio of % values directly.
No need to compute actual values.

Example:
North region (Total = ₹800 Cr): Product A = 35%, Product B = 25%.
Ratio A:B = 35:25 = 7:5. (No need to compute 280:200.)

ACROSS different rows OR different columns (different underlying totals):
MUST compute actual values before forming ratio.

Example:
North (Total=800): Product A = 35% → A_North = 280.
South (Total=600): Product A = 20% → A_South = 120.
Ratio = 280:120 = 7:3. (NOT 35%:20% = 7:4 — WRONG.)

SHORTCUT TEST: Are the totals the same? YES → use %. NO → compute values.
```

---

### 🚀 TRICK 14 — The +x% then −x% Net Change Trap

```
A value that increases by x% and then decreases by x% NEVER returns to original.
Net effect = −(x²/100)% (always a net DECREASE).

Proof: Final = Start × (1 + x/100) × (1 − x/100) = Start × (1 − x²/10000).

Examples:
+20% then −20%: Net = −(400/100)% = −4%.
+25% then −25%: Net = −(625/100)% = −6.25%.
+10% then −10%: Net = −(100/100)% = −1%.

Table DI version:
Production 2020 = 400. It increased 25% in 2021 and decreased 25% in 2022.
2021 = 400 × 1.25 = 500. 2022 = 500 × 0.75 = 375.
Net change from 2020 to 2022 = [(375−400)/400] × 100 = −6.25% ✓.

SPECIAL CASE: +25% then −20%:
1.25 × 0.80 = 1.00 → Net = 0% (exact break-even). NOT −5%.
Always multiply the multipliers to find the net effect.
```

---

### 🚀 TRICK 15 — Unit Conversion at the Last Step

```
RULE: Perform ALL calculations in the unit given in the table.
      Apply unit conversion ONCE at the very end.

Example:
Table unit: ₹ Lakhs.
Question: "What is the total revenue in ₹ Crores?"

WRONG: Convert each cell to Crores → add Crore values.
RIGHT: Add all Lakh values → divide the final SUM by 100.

Why RIGHT is right:
Wrong method = many conversions + accumulation of rounding errors.
Right method = one conversion, one rounding, one step.

Conversion chain to memorize:
Hundreds   → Thousands : ÷ 10
Thousands  → Lakhs     : ÷ 100
Lakhs      → Crores    : ÷ 100
Crores     → Lakhs     : × 100
Millions   → Crores    : × 10    (1 Crore = 10 Million)
Billions   → Crores    : × 100   (1 Billion = 100 Crores)
```

## 14. COMMON MISTAKE ALERTS

---

### ⚠️ MISTAKE 1 — Reading the Wrong Cell (Right Row, Wrong Column)

```
Wrong: In a 5-column table, reading column 3 when column 4 is needed.
       Especially dangerous in nested tables with sub-columns.

Right: State "Row = [X], Column = [Y]" before reading.
       Trace to that exact intersection — don't assume.

Prevention: The Cell Address System (Trick 1). Zero exceptions.
```

---

### ⚠️ MISTAKE 2 — Using the Wrong Base in % Change

```
Wrong: "% change from 2019 to 2022" using 2022 as the base.
Right: Old = 2019 (earlier year) = Base (denominator).
       % change = [(2022 value − 2019 value) / 2019 value] × 100.

Memory anchor: Earlier year = Old = Base. Always.
```

---

### ⚠️ MISTAKE 3 — Confusing Row Average with Column Average

```
"Average sales of Company A" = Average of Company A's COLUMN (across years).
"Average sales in 2021" = Average of the 2021 ROW (across companies).

These compute from different directions and give different numbers.
Wrong direction = completely wrong result.
```

---

### ⚠️ MISTAKE 4 — Forgetting to Convert Units

```
Table unit: ₹ Lakhs. Question asks for ₹ Crores.
Wrong: Report the Lakh value as Crores.
Right: Compute in Lakhs → divide by 100 → report in Crores.

Always: Identify the table unit at Layer 4 reading. Keep it in mind throughout.
```

---

### ⚠️ MISTAKE 5 — Confusing "% more than" with "% of"

```
A = 400, B = 300.
"A is what % of B?" → (400/300)×100 = 133.33%.
"A is what % more than B?" → (100/300)×100 = 33.33%.

133.33% ≠ 33.33%. Different questions. Different formulas.
Read the exact phrasing before choosing a formula.
```

---

### ⚠️ MISTAKE 6 — Comparing % Across Different Totals

```
In a percentage table:
North (Total=800): Product A = 35%.
South (Total=600): Product A = 20%.

"A_North is what % more than A_South?"
Wrong: [(35−20)/20]×100 = 75%.
Right: A_North = 280, A_South = 120. [(280−120)/120]×100 = 133.33%.

Different totals → different actual values → must compute values before comparing.
```

---

### ⚠️ MISTAKE 7 — Not Computing Derived Values for ALL Rows

```
"In which year was % profit margin maximum?"
Wrong: Compute % margin only for 2021 and 2022 (guessing they're the highest).
Right: Compute profit AND % margin for EVERY row, then compare.

Skipping rows = missing the actual maximum.
```

---

### ⚠️ MISTAKE 8 — Applying Reverse % Incorrectly

```
"2023 Revenue = ₹480 Cr, which is 20% more than 2022. Find 2022."
Wrong: 480 − (20% of 480) = 480 − 96 = 384. WRONG.
Right: 2022 = 480 / 1.20 = 400. CORRECT.

The 20% is relative to 2022 (Old), not 2023 (New).
Old × 1.20 = New → Old = New/1.20.
```

## 🏆 TOP 15 EXAM TIPS

1. **State the exact cell address before reading it** — "Row: 2021, Column: Company B → Revenue." This 3-second habit is the single most effective way to prevent the #1 error in table DI: right row, wrong column (or vice versa).

2. **Compute ALL missing cells before reading Question 1** — use row totals, column totals, given averages, or ratio/% clues. The missing cell is almost always directly asked about in at least one question.

3. **"Total of entity X over all years" = Column sum. "Total in year Y" = Row sum** — get this direction right every single time. Wrong direction = wrong calculation across every question.

4. **Base in % change = ALWAYS the chronologically earlier (Old) year** — no matter how the question is phrased. "From 2018 to 2022" → Old = 2018. "Growth of 2022 compared to 2018" → Old = 2018. One rule. No exceptions.

5. **"% more than B" → base is B. "% of B" → base is B. Whatever comes after "than" or "of" = denominator** — this Golden Rule covers every % question type. Test yourself: who is the base? Then apply the correct formula.

6. **Max average column = max total column (when row counts are equal)** — compare column totals directly. Save one division per column. Works for any equal-sample comparison.

7. **For derived values (profit, balance), compute the derived column for ALL rows first, then answer all derived-value questions** — don't compute profit only for the year being asked. You likely need to compare across years, so computing all rows upfront is faster.

8. **+x% then −x% gives a net DECREASE, not zero** — the formula is −(x²/100)%. For ±20%: net = −4%. For ±25%: net = −6.25%. This is a guaranteed trap question. Remember: the second % applies to the updated value, not the original.

9. **In percentage tables: NEVER compare % across rows with different totals** — compute actual values first. 35% of ₹800 Cr ≠ 35% of ₹600 Cr. The same % from different row totals gives different actual values.

10. **Grand total cross-check: sum of row totals = sum of column totals** — use this 15-second verification after computing totals. If they don't match, find the arithmetic error before answering questions.

11. **Reverse % calculation: "New is x% more than Old → Old = New / (1 + x/100)"** — never subtract x% from New. The x% is relative to Old (base), not New. This reversal appears in almost every exam.

12. **Pair complementary numbers for fast column sums** — look for pairs that sum to 500, 1000, or other round numbers. (200+300=500) + (240+360=600) + 180 = 1280. Much faster than sequential addition.

13. **For max % change, use fraction comparison (cross-multiplication) instead of division** — write Rise/Old for each period, cross-multiply pairs. No calculator needed. No rounding errors.

14. **Unit conversion happens ONCE at the very last step** — never convert individual cells mid-calculation. Sum/compute everything in the table's unit, then convert the final answer. One conversion vs. many.

15. **In nested tables, confirm three coordinates: Row, Main Column, Sub-Column** — state all three before reading any cell value. "Row: 2021, Company B, Revenue sub-column = 500." This prevents reading Company A's data when Company B is needed.
